User Pain Points

● Underwriter
  • No visibility into which observations trigger a policy failure — evaluation is a black box
  • Can't model "what if I apply this mitigation?" before recommending costly remediation to a client
  • No distinction between a partial fix (bridge) and a permanent resolution (full) — both look the same
  • No spatial representation of the property — risk is abstract, not visual
● Applied Scientist
  • Rule changes require an engineer — no self-service lifecycle (draft → review → activate)
  • No safe way to test a new rule against real observations before pushing it live
  • No audit trail for who changed a rule, when, or why
  • Historical queries impossible — can't reconstruct which rule version was active for a past policy period

Key Solution Decisions

Rule Data-Type Shape
✕
Flat columns per condition — separate SQL columns for field, operator, value. Simple but incapable of nesting or multi-step logic.
✕
Single unified JSON schema — one fixed JSON shape for all rule types. Forces every rule to carry fields it doesn't use; schema becomes a kitchen sink.
✓
Typed JSON blob + discriminator — a type string selects the evaluator class; definition JSON varies freely per type. Enables recursive condition trees (boolean_condition) and multi-item threshold logic (threshold_condition) without schema coupling.
Database Selection
SQLite — Chosen Zero-config, single-file, embedded. No server process. Fits the single-instance, local-first deployment model. Evaluation read-load is absorbed by the in-process rule cache, so DB is only hit on writes and historical queries.
PostgreSQL — Rejected Justified at scale (concurrent writers, multi-instance deploy) but adds operational overhead (server, migrations tooling) that provides no benefit here. The tradeoff reverses if the engine moves to a multi-tenant SaaS model.
Extensibility
New Rule Types Adding a condition type = one new evaluator class (subclass BaseEvaluator) + one entry in the EVALUATORS dispatch dict. Zero changes to the DB schema or existing evaluators.
Rule Versioning Rules are versioned by slug — multiple rows, one activated at a time. A new version activates atomically while the old one is deactivated in the same transaction. Historical queries reconstruct any past ruleset using epoch-ms timestamps.
Mitigation Types Full vs. bridge is a first-class distinction. Bridge mitigations carry modifier_params JSON consumed by the evaluator; full mitigations are purely declarative. New modifier strategies slot in without touching card rendering logic.
